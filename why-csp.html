Why your site should use a Content Security Policy

A Content Security Policy (CSP) is an HTTP response header that tells browsers which resources are allowed to load and execute on your pages. It is one of the most effective defenses against cross-site scripting (XSS) and related injection attacks.

What went wrong without a CSP?

When we looked up the URL you provided, no Content-Security-Policy header or equivalent HTML <meta> tag was found. That means browsers are not receiving explicit instructions about which scripts, styles, images, or connections are permitted on that site.

Without a policy, a single injected script tag or event handler can often run with the same privileges as your own code, potentially stealing session data, modifying page content, or redirecting users.

Security advantages of a CSP

  • Reduces XSS impact. CSP limits where scripts and styles may load from, blocking many inline and attacker-hosted payloads by default.
  • Controls third-party resources. You define trusted domains for scripts, fonts, images, frames, and network connections.
  • Restricts dangerous APIs. Policies can limit eval(), inline event handlers, and other high-risk execution paths.
  • Helps prevent clickjacking. The frame-ancestors directive specifies which sites may embed your pages in frames.
  • Supports monitoring. Report-only policies and reporting directives let you test changes and detect violations before enforcing them.
  • Encourages safer patterns. Teams adopt nonces, hashes, and external scripts instead of widespread inline code.

How to get started

  1. Return to the CSP Builder and create a policy for your site.
  2. Start with Content-Security-Policy-Report-Only to collect violations without blocking users.
  3. Review reports, tighten sources gradually, and switch to enforcing mode.
